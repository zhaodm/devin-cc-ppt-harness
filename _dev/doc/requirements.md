# PPT-Harness 需求汇总

本文件记录用户对 PPT-Harness 系统提出的所有需求和要求，按主题分类整理。

---

## 一、系统定位与目标

1. 高度可移植、Agent 驱动的教材制作系统
2. 从课程纲要到高质量 16:9 HTML 演示文稿的自动化生产
3. 核心价值：节省用户在 PPT 版式美化/图表制作上的时间，用户只需用文字表达主题/目标/逻辑/数据

---

## 二、运行环境与平台兼容

1. 必须同时支持三个平台：
   - Claude Code CLI（终端）
   - VSCode Cline 插件
   - VSCode Claude Code 插件对话框
2. 通过 CLAUDE.md + .clinerules + .mcp.json 实现跨平台规则同源
3. Agent 基于对话框输入斜杠命令触发，不依赖平台特定机制

---

## 三、流程与命令

1. 四个斜杠命令作为入口：
   - `/ppt-init` → 初始化 + 需求澄清
   - `/ppt-propose` → 需求分析 + 架构设计 + 测试用例
   - `/ppt-apply` → 开发 + 审计 + 人工审批
   - `/ppt-archive` → 归档 + 结项
2. 严格顺序执行：propose → apply → archive，禁止跳步
3. 需求编号自动生成，递增（REQ001, REQ002...）
4. 每步结束返回 PM，PM 检查通过后才启动下一步
5. 禁止跳过人工审批节点

---

## 四、角色隔离

1. 四个 Agent 角色：PM、SA、DE、TE
2. PM：只做调度、检查、人机交互，禁止技术判断/开发/设计/测试
3. SA：只做需求分析、架构设计，禁止开发编码、调度
4. DE：只做编码实现，禁止设计、需求定义、调度
5. TE：只做审计验证，禁止开发、设计、调度
6. 角色间传递必须经 PM 中转

---

## 五、Handoff 协议（角色隔离强化）

1. 采用纯文件协议 + 强化行为约束方案（跨平台兼容）
2. 角色切换必须通过 Handoff 文件实现，禁止仅靠对话文本传递
3. PM 切换角色前必须：写入 handoff → 更新 .state.md → 发出调度指令
4. 非 PM 角色仅读取 handoff 白名单中的文件
5. 非 PM 角色禁止引用对话历史中其他角色的推理或产出
6. 非 PM 角色完成后仅报告文件路径，不展开产物内容
7. 断点恢复时 PM 仅依据 .state.md，禁止依赖对话历史
8. Handoff 文件不可修改，重试创建新文件（追加轮次后缀）

---

## 六、视觉风格

1. 4色深色调色板：#1e2a3a（底色）、#ffffff（主文字）、#c8d6e5（辅助）、#e67e22（强调）
2. 正文字号 18px 起步，卡片标题 22px
3. 高信息密度：结构化布局，每页含明确结论
4. 每页需有图形元素（图标/图表/插图），禁止纯文本页面
5. 相邻页面禁止使用相同布局，追求视觉节奏感
6. 图标使用 Lucide Icons，按需从 CDN 下载 SVG
7. 完整视觉规范沉淀在 skills/design-visual.md

---

## 七、页面规范

1. 16:9 比例（1280x720px）
2. 每个章节作为单独的 HTML 页面，放在 pages/ 子目录
3. 通过 index.html 串联各章节，做好跳转逻辑
4. DE 拥有完全布局自由，不强制三段式结构
5. 每页必须有明确标题（h1 或等效）
6. 支持多种布局：英雄大图、左右分栏、时间线、卡片网格、对比表格等
7. 支持 Mermaid 图表
8. 支持键盘导航（← → 翻页 + .step 渐进揭示，Esc 返回索引）
9. 支持 .step 类分步展示（渐进式揭示内容）

---

## 八、开发模式

1. DE 强制 TDD 模式：编写测试(FAIL) → 实现代码(PASS) → 重构 → 自检
2. TE 必须通过真实浏览器（Playwright）检查显示效果
3. 审计失败最多 5 轮修复，超过上升到人工审核
4. apply 阶段逐页循环：每个页面必须走完 DE开发→TE审计→人工检查 后才能开始下一个页面
5. 逐页人工检查是轻量确认（该页OK），不是 SR2
6. 所有页面开发+审计+人工检查完成后，统一进行 SR2 正式审批 → DE合并 → TE最终审计 → SR3 人工审批

---

## 九、场景检测与变更模式

1. /ppt-init 启动时自动检测三种场景：
   - **NEW**：无历史需求，全新项目，执行完整需求澄清
   - **RESUME**：检测到未完成的 REQ（.state.md phase ≠ done），提示用户继续或放弃
   - **CHANGE**：有已完成的历史需求（spec/ 下有文件或有 phase=done 的 REQ），进入变更模式
2. RESUME 模式：不创建新 REQ，提示用户输入对应阶段命令继续
3. CHANGE 模式行为：
   - 自动备份当前 spec/ 到 spec/baselines/*.vN.md（版本号递增）
   - 创建新 REQ-ID 目录
   - 需求澄清仅围绕变更点（新增/修改/删除），不重复讨论未变更内容
   - 读取现有 spec/requirement-spec.md 和 spec/design.md 作为基线参考
4. 基线备份管理：
   - 每次进入 CHANGE 模式时自动备份
   - 备份路径：spec/baselines/{filename}.v{N}.md
   - 版本号从已有备份中取最大值 +1
5. 归档时 merge 而非覆盖：
   - 首次归档（spec/ 为空）：直接复制
   - 变更归档（spec/ 已有文件）：将变更内容 merge 到现有 spec 文件
   - 新增页面追加到页面清单，修改页面更新对应段落，删除页面标记移除

---

## 十、断点续作与 Token 节流

1. 支持自动识别已生成页面并跳过
2. 每完成一个页面的开发+审计后，清洗上下文中的 HTML 代码记忆
3. 只保留文件路径引用
4. 下一页开发时重新读取 design.md 对应段落，不依赖历史代码

---

## 十一、输出目录结构

1. 最终产物目录结构：output/final/{index.html, shared/, pages/}
2. index.html 位于顶层，作为首页/目录
3. 各章节页面放在 pages/ 子目录中
4. shared/ 存放 CSS/JS/图标/图片等共享资源
5. 页面中资源引用路径：../shared/（从 pages/ 子目录引用）
6. index.html 中链接指向 pages/*.html

---

## 十二、MCP 工具集成

1. zai-mcp-server：图片内容识别（stdio 模式，npx @z_ai/mcp-server）
2. web-search-prime：联网搜索（HTTP，智谱 BigModel API）
3. web-reader：网页内容抓取（HTTP，智谱 BigModel API）
4. 环境变量 Z_AI_API_KEY 统一鉴权

---

## 十三、实施阶段划分

用户要求按以下三阶段实施（非按功能模块拆分）：

1. 第一阶段：骨架搭建（基础设施）— 目录结构、Rules、Agent 契约、Skill 骨架、脚本入口、HTML 模板骨架
2. 第二阶段：流程贯通（Skills + Workflow）— Skill 执行细节、Workflow 串联、硬校验脚本
3. 第三阶段：质量打磨（模板 + 审美 + 断点续作）— 模板精修、Playwright 接入、断点续作、Token 节流、键盘导航

---

## 十四、产物纪律

1. reference/、spec/、output/final/ 只读，未经人工确认禁止修改
2. 每次写入产物后自检：文件存在 + 非空 + 格式合规
3. 禁止删除或重命名已有产物
4. .handoff/ 目录文件为过程记录，归档时保留用于审计追溯

---

## 十五、PM 心跳与过程日志

1. PM 在执行任何一条调度任务之前，必须打印心跳信息，格式: `[PM] xxx`
2. 心跳时机：调度前、验证后、审批前后、异常处理时、流程开始/结束时
3. 整条任务链所有角色的过程日志保存到 deliverables/{REQ-ID}/process.log
4. 日志格式: `[{时间}] [{角色}] {事件描述}`
5. PM 调度前/验证后各写一条，SA/DE/TE 完成后写一条，审批和异常各写一条
